Що таке ТЗ Tool і навіщо він потрібен

Клієнт надсилає пакет файлів: бриф у Word, креслення DWG, референси PDF, фотографії, таблиці Excel. Вручну читати все це, зіставляти між собою і складати ТЗ займає 1–3 години на один проект.

ТЗ Tool робить це автоматично: завантажуєш файли → натискаєш одну кнопку → отримуєш структуроване ТЗ з вимогами по кімнатах, планом роботи по стадіях, виявленими суперечностями і списком питань до клієнта.

2
Аналіз — кнопка "Розібрати файли"
🤖
Основний аналіз
Claude читає всі файли разом, зіставляє бриф з кресленнями, референси з коментарями. Витягує вимоги, знаходить суперечності, будує роадмап і чеклист здачі.
🔗
Автопошук посилань (Tavily)
Якщо введений Tavily-ключ — автоматично шукає посилання на меблі, матеріали і бренди. Запускається паралельно після основного аналізу.
📋
Побудова SOW-матриці
Після основного аналізу автоматично запускається другий легкий запит — перевіряє покриття кожного пункту SOW-шаблону. З'являється у вкладці ЗВІТ.
3
Перегляд результатів — чотири режими
КІМНАТИ
По приміщеннях
Всі вимоги згруповані по кімнатах (Вітальня, Кухня, Загальне тощо). Зліва список кімнат, справа — вимоги по категоріях. Основний режим роботи.
СТАДІЇ
По виробничих стадіях
Ті ж вимоги але відсортовані по стадіях виробництва: Моделінг → Текстуринг → Світло → Камери → Пост → Видача.
ТАБЛИЦЯ
Зведена таблиця
Всі вимоги в одній таблиці з фільтрацією і сортуванням. Колонки: Тип · Вимога · Цитата · Категорія · Кімната · Стадія · Джерело. Можна скачати як XLS.
ЗВІТ
SOW-матриця
Таблиця покриття по кожному пункту SOW-шаблону: ✅ знайдено / ⚠️ неповно / ❌ відсутнє. Джерело клікабельне — відкриває оригінальний файл на потрібній сторінці.
4
Секції в боковій панелі (режими Кімнати / Стадії)
⚠
SOW — Delivery Spec + Питання до клієнта
Delivery Spec — таблиця технічних параметрів здачі: роздільність, DPI, формат, час доби тощо. Кожен параметр має статус: ✓ з брифу (клієнт вказав явно), дефолт (підставляємо стандартне значення), ⚠ уточнити (невідомо).

Питання до клієнта — список пунктів що відсутні або незрозумілі. Для пунктів з дефолтом формулювання м'яке: "не вказано, буде підставлено дефолт X — підтвердіть або надішліть заміну". Є кнопка Запит — копіює список в буфер для відправки клієнту.
⚡
Конфлікти між файлами
Автоматично виявлені суперечності між документами. Наприклад: бриф каже "темно-сірі стіни", а мудборд показує світлий інтер'єр. Для кожного конфлікту вказані обидва джерела і конкретне питання для уточнення.
▶
Роадмап
Впорядкований план роботи по стадіях виробництва. Для кожної стадії — конкретні задачі, нотатки про залежності і ризики. Формується автоматично на основі всіх вимог.
✓
Чеклист здачі
Плоский список всіх вимог по стадіях — для звірки результату перед здачею клієнту. Моделер, текстурник і ПМ використовують один документ.
📋
Джерела
Посторінковий журнал — що знайдено в кожному файлі на кожній сторінці. Клік по тегу відкриває DocViewer на потрібній сторінці. Зручно для звірки: "звідки взялась ця вимога?"
💬
Коментарі клієнта
Всі рукописні нотатки, підписи і коментарі з файлів — зібрані в одному місці дослівно з прив'язкою до сторінки.

Перегляд джерел — DocViewer
↗
Кнопка "Джерело" у кожній вимозі
Натисни ↗ Джерело поряд з будь-якою вимогою — відкриється DocViewer на тій сторінці PDF де ця вимога знайдена. Навігація по сторінках всередині файлу.
🔗
Посилання на меблі і матеріали
Якщо Tavily знайшов посилання — вони відображаються прямо в картці вимоги. Відкриваються в системному браузері.
8
Експорт
📊 Excel (XLS)
Кнопка у тулбарі режиму Таблиця. Скачує tz-YYYY-MM-DD.xlsx. Колонки: Тип · Вимога · Цитата · Категорія · Кімната · Стадія · Джерело · Посилання. Посилання клікабельні в Excel. Враховує активні фільтри.
🖨 PDF (друк)
Кнопка PDF у хедері. Використовує window.print() з оптимізованими стилями друку: таблиця без розривів, UI-елементи приховані, посилання виводяться як текст.
