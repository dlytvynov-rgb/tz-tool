# Roadmap: Умная обработка ТЗ для 3D-визуализации

> **Цель:** Превратить любой входящий пакет файлов от клиента в структурированное, непротиворечивое ТЗ — понятное исполнителю, ПМу и LLM.

---

## Принцип

**ТЗ считается готовым к производству, если после его чтения можно начать моделинг без звонка клиенту.**

---

## Pipeline: 8 этапов

### Этап 1 — Приём и классификация входных файлов

Сортируем не по типу файла (PDF/JPG/DWG), а по типу **данных внутри**:

| Категория | Примеры |
|---|---|
| Планировки | floor plans, схемы расстановки |
| Фасады / разрезы | чертежи фасадов, сечения |
| Стиль / мудборд | референс-фото, pinterest, брендбук |
| Материалы и отделка | спецификации, каталожные позиции |
| Мебель / предметы | списки, арт-каталоги, ссылки |
| ТЗ текстом | Word, PDF с описанием задачи |
| Техтребования | форматы выдачи, сроки, кол-во ревизий |

**Выход:** каждый файл получает метку категории + флаг "распознан / не распознан"

---

### Этап 2 — Определение типа проекта и загрузка SOW-шаблона

Определяем тип проекта:

- Интерьер жилой
- Интерьер коммерческий (офис / ресторан / отель / ритейл)
- Экстерьер / фасад
- Мастерплан / генплан
- Продуктовая визуализация

Подгружаем SOW-шаблон для этого типа — это эталон того, **что должно быть** в полном ТЗ.

---

### Этап 3 — Валидация и детектор противоречий

**3a. Сравнение с шаблоном**
- Что есть из обязательного
- Чего не хватает
- Что есть, но непонятно / неполно

**3b. Детектор противоречий**
Явные флаги когда:
- Референсы по стилю конфликтуют друг с другом
- Планировка не вмещает заявленные зоны/предметы
- Заявленный стиль не соответствует бюджету/типу объекта
- Одна поверхность указана с разными материалами в разных файлах

---

### Этап 4 — Запрос недостающего + дефолты по типу проекта

Формируем **client request** — список что нужно донести клиенту.

Для каждой пустой позиции предлагаем **дефолт по типу проекта**:

```
Не указано: время суток для экстерьера
→ Дефолт для "Экстерьер жилой": "Золотой час, ~17:00, ясная погода, лето"
→ Если не подтвердите — используем это
```

Клиент получает: "вот чего не хватает + вот что будет, если не ответите"

---

### Этап 5 — Парсинг по производственным стадиям

| Стадия | Что нужно из ТЗ |
|---|---|
| **Моделинг** | Планировки с размерами, геометрия, что строить |
| **Текстуринг** | Материалы, отделка, бренды, цвета, RAL/артикулы |
| **Свет** | Время суток, сезон, источники света, атмосфера |
| **Камеры** | Ракурсы, высота глаза, ориентиры в кадре |
| **Пост-продакшн** | Стиль обработки, наличие людей, графические элементы |
| **Выдача** | Форматы, разрешение, кол-во ревизий, дедлайн |

---

### Этап 6 — Нормализация в единый формат

Всё входящее конвертируется в Markdown:
- PDF → текст + список изображений
- Изображения → вытащены отдельно + описание через vision
- Чертежи DWG/DXF → превью + ключевые размеры
- Excel → таблицы

**Выход:** единый `.md` документ — структурированное ТЗ, читаемое человеком и LLM.

---

### Этап 7 — Финальная структура ТЗ

```
1. О проекте       — тип, адрес/контекст, краткий бриф
2. Что рисуем      — список помещений / ракурсов / объектов
3. Планировки      — размеченные, с размерами
4. Стиль           — мудборд, описание атмосферы, что НЕ нравится
5. Материалы       — по зонам/поверхностям, с брендами
6. Мебель / декор  — список или "на усмотрение студии"
7. Свет            — время суток, сезон, искусственный свет
8. Камеры          — ракурсы, высота глаза, ориентиры
9. Техвыдача       — разрешение, формат, кол-во ревизий, дедлайн
10. Приоритеты     — что важнее всего если нужно срезать углы
```

---

### Этап 8 — Отчёт для ПМа и клиента

**Для ПМа (внутренний):**
- Таблица: пункт ТЗ / статус / источник / стадия
- Список противоречий с описанием
- Список вопросов к клиенту

**Для клиента (внешний):**
- Что получено и принято в работу
- Что заполнено дефолтом — нужно подтверждение
- Что критически отсутствует — нужно прислать

---

## Критерии готовности ТЗ

- [ ] Все обязательные поля по SOW заполнены (или одобрены дефолты)
- [ ] Нет неразрешённых противоречий между файлами
- [ ] Каждая производственная стадия имеет входные данные
- [ ] Клиент подтвердил дефолты или прислал замену
- [ ] Документ в едином Markdown-формате

---

## Приоритет разработки

| # | Фича | Приоритет |
|---|---|---|
| 1 | Классификация входных файлов по типу данных | Критично |
| 2 | SOW-шаблоны по типу проекта + валидация | Критично |
| 3 | Детектор противоречий | Высокий |
| 4 | Дефолты по типу проекта + client request | Высокий |
| 5 | Парсинг по производственным стадиям | Высокий |
| 6 | Конвертация всего в Markdown | Средний |
| 7 | Отчёт для клиента / ПМа | Средний |
| 8 | История сессий и переиспользование ТЗ | Низкий |
